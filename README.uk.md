# Система керування некритичними споживачами при зникненні центральної мережі

**[English](README.md) · Українська**

---


Автономна система на скриптах Shelly (Gen2+), яка вимикає некритичні потужні споживачі при зникненні центральної електромережі та повертає їх після стабільного відновлення. Працює напряму між пристроями (peer-to-peer по локальному HTTP), без MQTT-брокера і без хмари.

---

## 1. Призначення

Коли зникає центральна мережа, домогосподарство переходить на живлення від інвертора (акумулятора). Щоб зберегти заряд для критичних навантажень, некритичні потужні споживачі (бойлер, насос, тепла підлога тощо) потрібно автоматично вимкнути, а після стабільного повернення мережі — знову увімкнути.

Система складається з двох скриптів, що працюють у парі.

**[master.mjs](master.mjs)** — встановлюється на один пристрій Shelly (Master). Детектує стан центральної мережі і розсилає команди виконавцям. Детекція ведеться двома незалежними каналами:

- *Input-детектор* — стан цифрового входу, до якого підключений сухий контакт контактора наявності мережі.
- *Voltage-детектор* — якість напруги з гістерезисом: вимикає при виході за широкий коридор, дозволяє вмикати лише після стабільного перебування у вузькому коридорі протягом заданого часу.

Рішення «мережа є» приймається тільки коли обидва детектори згодні (логіка AND). Якщо пристрій не має вимірювання напруги — працює лише Input. Якщо не підключений контактор — лише напруга.

**[slave.mjs](slave.mjs)** — встановлюється на кожен пристрій-виконавець (Slave), який керує некритичним споживачем. Приймає команди від Master, виконує локально і повертає підтвердження.

---

## 2. Ключові властивості

- **Прямий канал peer-to-peer** по локальному HTTP. Працює навіть коли керуючий додаток недоступний.
- **Двофазне підтвердження** (за аналогією MQTT QoS): Slave спершу підтверджує прийом команди, потім — фактичне виконання зі станом реле.
- **Ідемпотентність** — повторна команда з тим самим ідентифікатором не призводить до подвійного спрацювання.
- **Повтори і watchdog** — якщо Slave не відповів, Master повторює команду, потім фіксує помилку.
- **Послідовне відновлення** — після повернення мережі споживачі вмикаються по черзі з паузою, щоб уникнути одночасного пускового навантаження.
- **Усі налаштування у коді** — жодних зовнішніх залежностей, KVS не використовується.
- **Події через MQTT** — Master публікує всі рішення і помилки через NotifyEvent для відстеження керуючим додатком.

---

## 3. Обмеження

- **Максимум 3 Slave** на один Master. Обумовлено обмеженнями середовища виконання скриптів Shelly (паралельні HTTP-виклики, таймери, пам'ять). Для більшої кількості споживачів використовується керування через MQTT-брокер на рівні додатка.
- Підтримуються пристрої Shelly Gen2 і новіші (RPC-архітектура).
- Slave призначений для пристроїв з одним реле.

---

## 4. Приклад використання

**Сценарій:** будинок із сонячною електростанцією та інвертором. При зникненні мережі живлення не переривається, але треба зберегти заряд — вимкнути бойлер, насос басейну і теплу підлогу, залишивши холодильник, освітлення і роутер.

**Конфігурація:**

- Master — Shelly 1PM Gen4 на лінії з вимірюванням напруги; до його входу підключений сухий контакт контактора наявності мережі.
- Slave 1 — Shelly на бойлері (вмикається першим при відновленні).
- Slave 2 — Shelly на насосі басейну.
- Slave 3 — Shelly на теплій підлозі.

**Робота:**

1. Зникає мережа → контактор розмикається → Master через 5 секунд розсилає всім Slave команду OFF одночасно.
2. Бойлер, насос і підлога вимикаються. Кожен повертає підтвердження.
3. Мережа повертається → Master чекає 60 секунд стабільної напруги → вмикає бойлер, через 30 секунд насос, ще через 30 секунд підлогу.
4. Холодильник, світло і роутер увесь час працювали без втручання — вони не під керуванням системи.

## 5. Підключення контактора для детекції стану мережі на прикладі Hager ESC125 NO (з нормально відкритими контактами)

### Призначення вузла

Master має дізнаватись, чи присутня центральна мережа. Оскільки після зникнення мережі живлення будинку не переривається (працює інвертор), сам факт наявності напруги в розетці не свідчить про стан мережі. Тому потрібен окремий датчик саме на лінії центральної мережі — до входу в інвертор. Цю роль виконує контактор: його котушка «відчуває» мережу, а силовий контакт передає цей стан на вхід Shelly як сухий контакт.

### Схема підключення

**Котушка контактора (клеми A1, A2)** підключається до контрольованої лінії — центральної мережі до входу в інвертор. Підключення не фазозалежне, для прикладу візьмемо L на A1, N на A2.

- Мережа є → на котушці 230 В → контактор притягнутий → силовий контакт 1-2 замкнутий.
- Мережі немає → котушка знеструмлена → контакт 1-2 розімкнутий.

**Силовий контакт контактора (клеми 1, 2)** використовується як сухий сигнальний контакт:

- клема `1` → підключається до нуля `N` резервного живлення (того самого, від якого живиться Shelly);
- клема `2` → підключається до клеми `SW` реле Shelly (Master).

**Живлення самого Shelly (Master і всіх Slave)** — обов'язково від резервного джерела (виходу інвертора), а не від центральної мережі. Інакше при зникненні мережі знеструмиться і сам Shelly, і вся система перестане працювати саме тоді, коли вона потрібна.

### Чому на сигнальний контакт подається N, а не L

Вхід `SW` у Shelly Gen4 визначає свій стан за наявністю електричного потенціалу на ньому відносно власного живлення пристрою. Поширене непорозуміння — вважати, що на `SW` обов'язково треба подавати фазу `L`. Для **керування навантаженням зовнішнім вимикачем** так і роблять: вимикач пропускає фазу на `SW`. Але в нашому випадку завдання інше — не керувати, а лише **зчитати, замкнутий контакт чи ні**.

Для зчитування стану контакту неважливо, який саме провідник через нього замикається. Замикання нуля `N` через сухий контакт контактора дає той самий результат `input:0 state:true`, що й замикання фази. Тому ми використовуємо `N` з трьох причин:

1. **Безпека.** На сигнальному проводі від контактора до Shelly немає фази. Дотик до цього кола під час монтажу чи обслуговування безпечніший.
2. **Силові контакти реле Shelly не задіяні.** Ми використовуємо лише вхід `SW`. Вихідні силові контакти реле Shelly (`O`/`I`) залишаються вільними — на Master вони не потрібні, бо Master нічим не комутує, а тільки детектує і керує іншими по мережі.
3. **Мінімальне навантаження на контакт контактора.** Через вхід `SW` тече лише мікрострум (міліампери), а не робоче навантаження. Контакт контактора працює в найлегшому режимі, його ресурс практично не витрачається.

### Логіка роботи

| Стан мережі | Котушка A1-A2 | Контакт 1-2 | Сигнал на SW | `input:0` | Рішення Master |
|---|---|---|---|---|---|
| Є | під напругою | замкнутий | N приходить | `state:true` | grid_present=true |
| Немає | знеструмлена | розімкнутий | нічого | `state:false` | grid_present=false |

### Примітка щодо вибору контакту

ESC125 має один силовий контакт 1NO (клеми 1-2), розрахований на 25 А. Для детекції цього достатньо. Якщо ж силовий контакт на вашому контакторі вже використовується для комутації реального навантаження, для сигналу потрібен окремий допоміжний контакт — у такому разі оберіть модель контактора з двома каналами, наприклад Hager ESC225. Подавати на `SW` сигнал від уже навантаженого силового контакту не можна.

## 6. Інструкція зі встановлення та налаштування

### Крок 1. Підготовка пристроїв

Переконайтесь, що всі пристрої Shelly:

- покоління Gen2 або новіше;
- оновлені до актуальної прошивки;
- підключені до однієї локальної мережі;
- мають статичні IP-адреси (через DHCP-резервування на роутері або статичне налаштування в самому Shelly). Це критично — Master звертається до Slave за IP;
- живляться від резервного джерела (виходу інвертора).

### Крок 2. Монтаж контактора-детектора

Виконайте підключення контактора згідно розділу 5. Перед встановленням скриптів переконайтесь, що `input:0` на Master правильно змінює стан при вимкненні/увімкненні центральної мережі (це видно у вебінтерфейсі Master, розділ Components).

### Крок 3. Встановлення Slave (на кожному виконавці)

1. Відкрийте вебінтерфейс пристрою-виконавця за його IP.
2. Перейдіть у розділ **Scripts**.
3. Натисніть **Add script**.
4. Вставте код **slave.mjs**.
5. **Важливо:** скрипт має бути першим у списку, тобто мати **id = 1**. Якщо на пристрої вже є інші скрипти і цей отримає інший номер — або видаліть зайві скрипти, або змініть константу `SLAVE_SCRIPT_ID` у Master відповідно.
6. Налаштуйте константи зверху файлу за потреби:
   - `SW_ID` — номер каналу реле. Для пристрою з одним реле залишіть `0`.
   - `DEADMAN_SEC` — страховка. `0` означає вимкнено. Якщо хочете, щоб реле саме вимикалось через N секунд без нової команди від Master — поставте потрібне число.
7. Натисніть **Save**.
8. Увімкніть автозапуск: позначте **Run on startup**.
9. Натисніть **Start**.
10. У консолі має з'явитись `ms_slave_ready`.

Повторіть для всіх виконавців (до трьох).

### Крок 4. Перевірка доступності Slave

З браузера відкрийте на кожному Slave:

```
http://<IP_SLAVE>/script/1/cmd
```

Має повернутись помилка HTTP 405 (Method Not Allowed) — це нормально, бо браузер робить GET, а endpoint чекає POST. Якщо повертається 404 — скрипт не запустився або має інший id; поверніться до кроку 3.

### Крок 5. Встановлення Master (на детекторі мережі)

1. Відкрийте вебінтерфейс пристрою-детектора.
2. Перейдіть у **Scripts → Add script**.
3. Вставте код **master.mjs**.
4. Налаштуйте константи зверху файлу.

   **Список виконавців** — впишіть IP ваших Slave у потрібному порядку. Перший у списку вмикається першим при відновленні мережі:

   ```javascript
   let TARGETS = [
     "192.168.1.20",
     "192.168.1.21",
     "192.168.1.22"
   ];
   ```

   Якщо Slave один — залиште один рядок, решту видаліть або закоментуйте.

   **Пороги напруги** (за потреби під вашу мережу):

   ```javascript
   let V_LOW_OFF = 190;    // нижче цього — вимкнути
   let V_LOW_ON = 200;     // вище цього — дозволити вмикати
   let V_HIGH_OFF = 260;   // вище цього — вимкнути
   let V_HIGH_ON = 250;    // нижче цього — дозволити вмикати
   let V_STABLE_SEC = 60;  // скільки секунд напруга має бути стабільною
   ```

   **Таймінги:**

   ```javascript
   let DEBOUNCE_OFF_SEC = 5;   // затримка перед вимкненням (захист від коротких збоїв)
   let DEBOUNCE_ON_SEC = 60;   // затримка перед увімкненням (підтвердження стабільності)
   let ON_GAP_SEC = 30;        // пауза між вмиканням сусідніх Slave
   ```

5. Налаштуйте детектор мережі. У функції `inputPresent()` за замовчуванням читається `input:0`. Якщо ваш сигнал наявності мережі підключений до іншого входу або має інвертовану логіку — відредагуйте цю функцію.
6. Натисніть **Save**, позначте **Run on startup**, натисніть **Start**.
7. У консолі має з'явитись:

   ```
   ms_voltage_source {"source":"switch"}   ← або "em", "em1", "none"
   ms_started {"schema":"master-slave-v1","n_targets":1}
   ```

### Крок 6. Перевірка роботи

Слідкуючи за консоллю Master:

1. **Розімкніть** контактор/вхід наявності мережі (імітація зникнення мережі). Через ~5 секунд має з'явитись:

   ```
   ms_decision {"grid_present":false}
   ms_executed {...,"actual_on":false,...}   ← по одному на кожен Slave
   ```

   Споживачі мають вимкнутись.

2. **Замкніть** контактор/вхід (імітація повернення мережі). Через ~60 секунд:

   ```
   ms_decision {"grid_present":true}
   ms_executed {...,"actual_on":true,...}
   ```

   Споживачі мають увімкнутись по черзі.

Якщо `ms_decision` з'являється, а `ms_executed` ні — перевірте: IP у `TARGETS` правильні; Slave-скрипти запущені (крок 4); усі пристрої в одній мережі.

### Крок 7. Зміна налаштувань після встановлення

Будь-яка зміна (новий IP, інші пороги, таймінги) робиться редагуванням констант зверху відповідного скрипта і натисканням **Save**. Скрипт перезапускається автоматично, нові значення застосовуються одразу. Жодних додаткових дій не потрібно.

---

## 7. Події для інтеграції з додатком

Master публікує через MQTT (NotifyEvent у топік `<device_id>/events/rpc`):

| Подія | Значення |
|---|---|
| `ms_started` | скрипт запущений |
| `ms_voltage_source` | яке джерело напруги визначено (`switch` / `em` / `em1` / `none`) |
| `ms_decision` | змінився стан мережі (`grid_present`) |
| `ms_executed` | Slave підтвердив виконання (зі станом реле) |
| `ms_voltage_bad` / `ms_voltage_recovering` / `ms_voltage_good` | зміни якості напруги |
| `ms_send_err` / `ms_retry` / `ms_failed` | проблеми зв'язку з Slave |

Це дозволяє керуючому додатку відображати стан системи і отримувати сповіщення, не втручаючись у роботу автономного резервного контуру.

## 8. Протокол обміну (master-slave-v1)

Master → Slave, POST на `http://<slave_ip>/script/1/cmd`:

```json
{
  "cmd_id": "c1a2b3c4-...",
  "schema": "master-slave-v1",
  "grid_present": false,
  "ts": 1778938580,
  "ack_url": "http://<master_ip>/script/<id>/ack"
}
```

Slave → Master (синхронна відповідь у HTTP, фаза 1 — прийнято):

```json
{ "cmd_id": "...", "ack": "RECEIVED", "schema": "master-slave-v1" }
```

Slave → Master, POST на `ack_url` (фаза 2 — виконано):

```json
{
  "cmd_id": "...",
  "ack": "EXECUTED",
  "ok": true,
  "actual_on": false,
  "ts": 1778938581
}
```

Повторна команда з тим самим `cmd_id` повертає кешований результат із полем `"ack": "DUPLICATE"` — це забезпечує безпечні повтори без подвійного спрацювання реле.
